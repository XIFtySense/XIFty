# Plan #58: Audio phase 2 — M4A as ISOBMFF variant (iTunes udta/meta/ilst)

## Problem
Today, files with `ftyp` major brand `M4A `, `M4B `, or `M4P ` are detected as `Format::Mp4` (see `crates/xifty-detect/src/lib.rs:92-106` where `b"M4A "` is lumped into `mp4_brand`). The ISOBMFF container parser already walks `moov/udta/meta/ilst` (see `crates/xifty-container-isobmff/src/lib.rs:335-336` and `381-385`), but only surfaces three iTunes atoms (`©ART`, `©too`, `©nam`) via the tiny `xifty-meta-quicktime` decoder (`crates/xifty-meta-quicktime/src/lib.rs:1-78`). Per parent task #5 and the issue's ambiguity-#2 recommendation, we need `M4A` to be a first-class `Format::M4a` variant so (a) detection and capability reporting are honest, (b) iTunes `ilst` atoms round-trip into a normalized tag namespace, and (c) probe surfaces audio-only track facts (duration / sample rate / channels / codec) that the existing ISOBMFF layer already computes.

## Approach
Mirror the existing audio-format split (`Format::Mp3` → `xifty-container-id3` + `xifty-meta-id3v2`, wired through `crates/xifty-cli/src/lib.rs`). Add `Format::M4a` to `xifty-core`, branch on `ftyp` major brand in `xifty-detect` before the existing `is_mp4_brand` check (so `M4A /M4B /M4P ` route to `M4a` and MP4 continues to catch `isom/mp41/mp42/...`). Extend `xifty-container-isobmff` to emit the full set of iTunes four-char atoms under `ilst` as `IsobmffPayload { kind: "itunes", tag: Some(fourcc), ... }` while leaving the existing `quicktime` payload kind untouched (container vs. metadata separation: container surfaces raw atom payloads, metadata crate interprets them). Create a new `xifty-meta-itunes` crate that interprets iTunes `data` atoms into `MetadataEntry` values in a dedicated `itunes` namespace. Wire `Format::M4a` through `probe_source` and `extract_source` in `xifty-cli` reusing `parse_isobmff` and `isobmff_entries`, adding a new `itunes_entries` helper. Add a synthetic M4A fixture (`happy.m4a`) to `tools/generate_fixtures.py` mirroring `build_media_file` with `major_brand=b"M4A "`, an audio-only track, and a richer `ilst` (covers `©nam/©ART/©alb/©day/©gen/trkn/disk/covr`). Update `CAPABILITIES.json` to add `m4a` container with `itunes: bounded`. No FFI change (JSON passes through unchanged).

## Risks
- **Snapshot churn on MP4/MOV fixtures**: Today `b"M4A "` is in `mp4_brand`. Removing it means any fixture with `M4A ` as a compatible brand that previously matched via `is_mp4_brand` may now miss. Mitigation: keep `M4A ` in the `compatible_brands` check within `is_mp4_brand` (only the *major brand* branch routes to `M4a`) — decision: `is_m4a_brand` inspects major brand only; `is_mp4_brand` stays as-is minus the `M4A ` entry. Confirm no existing fixture regresses by running the full test suite.
- **Dual-emit for iTunes atoms (`quicktime` + `itunes`)**: Short-term duplication keeps existing mp4/mov snapshots stable but could double-count in normalized fields. Mitigation: have `xifty-meta-itunes` produce entries only in the `itunes` namespace; normalization policy already dedupes via `dedupe_conflicts` (`crates/xifty-cli/src/conflict_dedupe.rs`), but inspect normalized output for duplicates.
- **`covr` atom shape variance**: Real-world `covr` atoms embed JPEG or PNG; flags field of the inner `data` box distinguishes. For the synthetic fixture, a minimal PNG stub keeps tests deterministic; for real files we expose `TypedValue::Bytes` with a note — no image decoding in v1.
- **`CAPABILITIES.json` regeneration**: If `tools/generate_capabilities.py` produces the file deterministically from crate metadata, manual edits may be overwritten. Verify before committing; if so, update the generator script's data source instead.
- **FFI stability**: Adding a new `Format` variant changes the JSON enum surface (`detected_format: "m4a"`). This is additive and matches the existing pattern for `mp3`; no C ABI shape change. Document in the PR body.
