# Plan #61: DNG verification against TIFF + optional DNG-specific tag pass

## Problem
DNG is structurally a TIFF variant and is claimed (at the roadmap level) to be partially covered by `xifty-container-tiff`. No DNG fixture or code path currently exists in the repo â€” `crates/xifty-detect/src/lib.rs` (lines 10â€“12) only emits `Format::Tiff` for `II*\0` / `MM\0*` headers, `crates/xifty-core/src/lib.rs` (lines 8â€“19) has no `Format::Dng` variant, and `CAPABILITIES.json` lists no `dng` container. The task is an `xs` verification-and-decision: probe a DNG fixture through the existing TIFF path, then choose between (a) flag as TIFF-equivalent with a `dng` capability tag, (b) add a detect branch on `DNGVersion` (tag `0xC612`) so it reports as `Format::Dng`, or (c) scope a follow-up if interpretation work is needed.

## Approach
Synthesize a minimal DNG fixture by extending `tools/generate_fixtures.py`'s existing `build_tiff` helper (line 38) to emit a TIFF with the DNG-required tags (`DNGVersion = 0xC612`, `DNGBackwardVersion = 0xC613`, `UniqueCameraModel = 0xC614`, plus a `SubIFDs = 0x014A` pointing to a stub raw IFD). Probe it through `xifty-cli` to verify the TIFF container parser walks it without errors. Given that DNG is deliberately TIFF-shaped and the task rubric flags outcome (2) as the expected middle path, add a lightweight `Format::Dng` detect branch keyed on the presence of `DNGVersion` in IFD0 â€” mirroring the `ftyp`-brand branching pattern already used in `is_heif_brand` / `is_mov_brand` (`crates/xifty-detect/src/lib.rs` lines 51â€“102). The container probe path in `crates/xifty-cli/src/lib.rs` adds a `Format::Dng` arm that reuses `parse_tiff` (mirroring the existing `Format::Tiff` arm at lines 51â€“54) and labels the container as `"dng"` so the snapshot is distinguishable. `CAPABILITIES.json` gains a `dng` container entry marked `bounded` with the same namespace matrix as `tiff`.

If probing the fixture reveals parser issues (outcome 3), the builder stops, documents findings in the issue thread, and scopes a follow-up â€” no code changes land.

## Step-by-step

2. Add `Format::Dng` to `xifty_core::Format` and `as_str` mapping â€” verifiable by `cargo check -p xifty-core`.
3. In `xifty-detect`, after the current TIFF-magic check (line 10), peek into IFD0 to check for tag `0xC612`; on match return `Format::Dng`, otherwise fall through to `Format::Tiff`. 5. Add snapshot tests `probe_happy_dng` and `extract_happy_dng_normalized` in `crates/xifty-cli/tests/cli_contract.rs`; accept snapshots via `cargo insta review`.
6. Update `CAPABILITIES.json` to include a `dng` container entry marked `bounded`, mirroring `tiff`. Run `python tools/generate_capabilities.py` if generation is driven from code; otherwise hand-edit and confirm hygiene check passes.
7. Post findings comment on issue #61 summarizing: fixture parses cleanly through TIFF path, DNG is now a distinct `Format::Dng`, capabilities reflect `bounded`. Confirm "no FFI shape change" â€” verified by `cargo test -p xifty-ffi --all-features` passing without snapshot/contract edits.

## Risks
- The IFD0 peek in `xifty-detect` must be defensive: malformed TIFFs could have an out-of-bounds first-IFD offset. Keep the probe read-only and fall back to `Format::Tiff` on any read error rather than bubbling a parse error out of detection â€” detection has never returned `Parse` before.
- Snapshot output for DNG will differ from TIFF only by the container label; reviewers must accept the new snapshot deliberately, not blanket-accept (SRS Â§4 snapshot drift policy).
- If the chosen fixture layout accidentally triggers `xifty-meta-exif` decoding of the DNG-private tags as unknowns, the `extract` snapshot may include noise. Keep the fixture minimal â€” only the three DNG marker tags plus the standard TIFF baseline â€” to keep snapshots stable.
- `CAPABILITIES.json` may be generated rather than hand-authored. If `tools/generate_capabilities.py` owns the file, edit the generator and regenerate; otherwise edit directly and verify `hygiene.yml` still passes.
- No FFI surface change is expected, but adding a `Format` enum variant is a behavioral surface change visible in probe JSON (`input.detected_format`). That is intended per the acceptance criteria and should be called out in the PR description.
