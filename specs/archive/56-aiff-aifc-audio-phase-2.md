# Plan #56: Audio phase 2 — AIFF/AIFC (IFF parser)

## Problem
XIFty currently covers audio only via the MP3/ID3v2 path delivered in Task 1 (`xifty-container-id3` + `xifty-meta-id3v2`). The roadmap in issue #5 (parent) lists AIFF as part of the P0 audio wedge — AI/music creator workflows deliver AIFF/AIFC alongside WAV/FLAC. Neither the `Format` enum (`crates/xifty-core/src/lib.rs:8-17`) nor the detector (`crates/xifty-detect/src/lib.rs:4-44`) recognizes the IFF `FORM` magic, and there is no container crate that walks the big-endian IFF chunk structure to expose `COMM` (audio params + frame count → duration), `SSND` (audio data offset), or the optional `ID3 ` chunk that many AIFF writers emit.

## Approach
Add a new workspace crate `xifty-container-aiff` modelled on `xifty-container-id3` (hierarchical `ContainerNode` emission, `Issue` propagation, absolute-offset accounting) and `xifty-container-riff` (chunk-walk shape, accessor helpers). The parser is container-only per `specs/SRS.md §3.1`: it decodes `COMM` (sample rate = 80-bit IEEE extended, num_channels u16 BE, sample_size u16 BE, num_sample_frames u32 BE) and locates `SSND` + `ID3 ` byte ranges, but delegates ID3v2 tag decoding to the existing `xifty-meta-id3v2` crate (mirroring the MP3 wiring at `crates/xifty-cli/src/lib.rs:972-1020`). Detection adds an IFF `FORM`+`AIFF`/`AIFC` magic branch alongside the RIFF/WEBP branch at `crates/xifty-detect/src/lib.rs:18-20`. The CLI probe/extract arms mirror the MP3 arm at `crates/xifty-cli/src/lib.rs:71-74` and `:467-472`, emitting `AudioSampleRate`/`AudioChannels`/`AudioBitDepth`/`DurationSeconds` entries with namespace `"aiff"` — the existing `xifty-policy` tag table already routes these tag names into `audio.sample_rate`/`audio.channels`/`audio.bit_depth`/`duration` for any namespace (`crates/xifty-policy/src/lib.rs:181-238`), so no policy work is required for normalization.

## Step-by-step
1. Add `crates/xifty-container-aiff/Cargo.toml` + `src/lib.rs` with `parse_bytes` that validates `FORM`+`AIFF`/`AIFC` magic, walks 8-byte chunks (`id:[u8;4]`, `size:u32 BE`, padded to even length), and emits `ContainerNode` entries with parent label `"aiff"` — `cargo test -p xifty-container-aiff` covers a synthetic minimal AIFF (COMM + SSND) and asserts form_type, chunk count, nodes.
2. Decode `COMM`: read `num_channels (u16 BE)`, `num_sample_frames (u32 BE)`, `sample_size (u16 BE)`, `sample_rate (80-bit IEEE 754 extended BE)` → f64; compute `duration_seconds = num_sample_frames / sample_rate` when both are finite and positive, otherwise `None` with an `Issue { code: "aiff_comm_invalid_rate" | "aiff_comm_missing", severity: Warning }` — unit tests cover the extended-float decoder (44100/48000/22050) and missing/short `COMM` paths.
3. Handle `SSND` (record `data_offset` for provenance only) and the optional `ID3 ` chunk (store `offset` + `len`, strip trailing pad byte before exposing via `id3v2_payload()`); non-AIFF/AIFC form types push `Severity::Info` issue `aiff_non_standard_form` — unit tests exercise SSND offset, ID3-present/absent branches, and an AIFC form variant.
4. Wire the crate into the workspace by appending `"crates/xifty-container-aiff"` to `Cargo.toml` members (lines 2-28) — `cargo build` succeeds.
5. Extend `Format` in `crates/xifty-core/src/lib.rs:8-17` with `Aiff` and add `Self::Aiff => "aiff"` to `as_str` — exhaustive `match`es in detect + cli (only consumers, verified via grep) compile after step 6/7.
6. Extend `crates/xifty-detect/src/lib.rs` with a block, after the RIFF/WEBP branch at lines 18-20, that returns `Format::Aiff` when `bytes[0..4] == b"FORM"` and `bytes[8..12] == b"AIFF"` or `b"AIFC"`; extend `detects_formats` at lines 128-184 with aiff + aifc fixtures — `cargo test -p xifty-detect` passes.
7. Add `xifty-container-aiff` to `crates/xifty-cli/Cargo.toml` `[dependencies]` and wire `Format::Aiff` branches into both `probe_source` (after line 74) and `extract_source` (after line 472), returning container name `"aiff"`, container nodes from the parser, and an `aiff_entries()` helper that (a) decodes ID3v2 via `decode_id3v2_payload` when `id3v2_payload()` is present and (b) emits `AudioSampleRate`/`AudioChannels`/`AudioBitDepth`/`DurationSeconds` entries with namespace `"aiff"` — `cargo build -p xifty-cli` passes.
8. In `tools/generate_fixtures.py`, add `build_aiff(variant)` that constructs FORM+AIFF with a COMM chunk (44100 Hz, 2 channels, 16-bit, N frames), SSND chunk with PCM bytes, and an optional ID3v2 tag chunk; register `"happy.aiff": build_aiff("with_id3")` and `"happy.aifc": build_aiff("aifc")` in the dict at lines 840-842 — running the script produces both files under `fixtures/minimal/`.

## Risks
- 80-bit IEEE 754 extended-precision decoder is easy to get subtly wrong — must test all three common rates plus zero-exponent and infinity/NaN edges, guarded by an `aiff_comm_invalid_rate` issue.
- AIFC extends `COMM` with `compression_type (u32)` and a `compression_name` Pstring after the sample rate — the parser must gate reads on chunk `size`, not fixed offsets, so plain AIFF (no compression fields) does not overrun.
- `ID3 ` chunk trailing-byte padding (IFF chunks pad to even length) — off-by-one risk when computing `id3_payload_len`; strip the pad byte before passing into `xifty-meta-id3v2`.
- Malformed `size` claiming > file length — mirror the `riff_chunk_length_invalid` guard at `crates/xifty-container-riff/src/lib.rs:118-131` and emit `aiff_chunk_length_invalid`.
- `tools/generate_fixtures.py` output must be deterministic so insta snapshots do not churn across CI runs — use fixed PCM samples and fixed ID3 payload sizes.
