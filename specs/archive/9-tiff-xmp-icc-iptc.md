# Plan #9: TIFF container — wire XMP (tag 700), ICC (tag 34675), IPTC (tag 33723)

## Problem
`CAPABILITIES.json` lines 42–48 mark TIFF's `xmp`, `icc`, and `iptc` as `not_yet_supported`. The CLI extraction path for `Format::Tiff` at `crates/xifty-cli/src/lib.rs:178–195` only runs `decode_from_tiff` (EXIF), `decode_apple_from_tiff`, and `decode_sony_from_tiff`. JPEG (same file, lines 97–176), PNG (196–260), and WebP (262+) branches already route XMP/ICC/IPTC payloads through `decode_packet`, `decode_icc_payload`, and `decode_iptc_payload`. Standalone TIFF files therefore silently lose XMP (TIFF tag 700 / 0x02BC, XMLPacket), ICC (tag 34675 / 0x8773, InterColorProfile), and IPTC-IIM (tag 33723 / 0x83BB). Acceptance criteria require wiring these three namespaces, preserving the container/interpretation crate boundary, and updating `CAPABILITIES.json`.

## Approach
**Option B chosen** (free functions over `(bytes: &[u8], &TiffContainer)`), mirroring the existing `decode_from_tiff(bytes, base_offset, container, &tiff)` signature already used across `xifty-meta-exif`, `xifty-meta-apple`, and `xifty-meta-sony`. Rationale: `TiffContainer` at `crates/xifty-container-tiff/src/lib.rs:16–21` stores only `endian / nodes / entries / issues` — no byte buffer — and every TIFF consumer in the workspace already threads the source byte slice separately (see `Format::Tiff` arm line 180 `source.bytes()`, PNG EXIF line 209 `payload`, WebP EXIF line 289 `payload`). Adding a `raw: Vec<u8>` field would be a bigger churn than needed and would duplicate bytes already held by the caller; Option B preserves the existing invariant.

Add three free functions in `xifty-container-tiff`:
- `xmp_payload(bytes: &[u8], tiff: &TiffContainer) -> Option<(u64, &[u8])>`
- `icc_payload(bytes: &[u8], tiff: &TiffContainer) -> Option<(u64, &[u8])>`
- `iptc_payload(bytes: &[u8], tiff: &TiffContainer) -> Option<(u64, &[u8])>`

Each walks `tiff.entries`, finds the first entry whose `tag_id` matches (0x02BC / 0x8773 / 0x83BB), computes `byte_len = value_size(type_id) * count`, and returns `(absolute_offset, slice)` using `value_offset_absolute` when `byte_len > 4`. For inline values (`byte_len <= 4`) the payload is packed into `value_or_offset`; since XMP/ICC/IPTC payloads are always large in practice, the inline branch returns `None` (documented; not worth the endianness reconstruction). Return `None` when `value_offset_absolute` is `None` or when the resolved slice would exceed `bytes.len()` — never panic.

CLI integration: extend the `Format::Tiff` arm at `crates/xifty-cli/src/lib.rs:178–195` to call each helper against `source.bytes()` and feed the returned slice into existing `decode_packet` / `decode_icc_payload` / `decode_iptc_payload`, pushing `namespace_issue` entries on empty decode — identical structure to JPEG lines 123–175. Provenance `path` strings are `ifd0_xmp`, `ifd0_icc`, `ifd0_iptc`. `CAPABILITIES.json` is promoted from `not_yet_supported` to `bounded` (matching JPEG). Fixtures are generated by extending `tools/generate_fixtures.py::build_tiff` (line 38) to accept optional payload kwargs; register three new fixtures near line 622. Insta snapshots cover the new paths.

## Step-by-step
1. In `crates/xifty-container-tiff/src/lib.rs`, add a private helper `payload_for_tag(bytes: &[u8], tiff: &TiffContainer, tag_id: u16) -> Option<(u64, &[u8])>` that (a) finds the first matching entry, (b) computes `byte_len = value_size(entry.type_id) * entry.count`, (c) when `byte_len > 4` and `entry.value_offset_absolute` is `Some(abs)`, translates `abs` back to a local index relative to the TIFF payload (using the base offset carried by existing callers — since these free functions are called with the same `bytes` slice that was passed to `parse_bytes`, the absolute offset equals the local index when `base_offset = 0`, and when `base_offset > 0` callers must pass the correct slice; the helper performs bounds checking `local + byte_len <= bytes.len()` before slicing). ## Risks
- **Option B base-offset semantics**: the free helpers take the raw TIFF byte slice and a `TiffContainer` parsed from those same bytes. `value_offset_absolute` is computed via `cursor.absolute_offset(local)` in `walk_ifd` (line 144), which adds `base_offset` passed to `parse_bytes`. For `Format::Tiff` the caller passes `source.bytes()` with `base_offset = 0`, so `value_offset_absolute == local_index`. The helper must subtract `tiff.nodes[0].offset_start` (the container root offset, which carries `base_offset`) before indexing into `bytes`. Documented in step 1.
- **Endianness of payloads**: ICC profiles are big-endian regardless of TIFF byte order; IPTC-IIM has its own framing. The helpers return raw bytes untouched — no byte-swapping.
- **Inline (≤4 byte) payloads**: Scoped out — XMP/ICC/IPTC payloads are always large in practice. Helpers return `None` for inline cases; a future issue may add inline reconstruction if a real fixture demands it.
- **Multiple occurrences**: Spec-wise each tag appears once in IFD0; in rare real files XMP may live in sub-IFDs. Helpers scan all `tiff.entries` (any IFD) and return the first match. Documented as a known limitation.
- **Fixture determinism**: `generate_fixtures.py` IFD ordering matters for snapshot stability — reuse the deterministic offset layout already used for `happy.tiff`.
- **CAPABILITIES promotion level**: `bounded` matches JPEG's levels for `xmp`/`icc`/`iptc` and the acceptance criterion ("at least `bounded`"); safer for v1 than `supported` since XMP/IPTC decoders are themselves `bounded`.
- **Malformed-fixture follow-up**: no `malformed_xmp.tiff` in this scope; the step-1 out-of-bounds unit test satisfies SRS §3.5 at the container layer. A `malformed_iptc.tiff` snapshot parallel to `malformed_iptc.jpg` is explicitly out-of-scope for this issue — tracked separately.
