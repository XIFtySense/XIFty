# Plan #55: Audio phase 1 — FLAC (native block parser + Vorbis comments)

## Problem
XIFty currently has no FLAC path. `Format` in `crates/xifty-core/src/lib.rs:8-17` enumerates only JPEG/TIFF/PNG/WebP/HEIF/MP4/MOV/MP3, and `detect()` in `crates/xifty-detect/src/lib.rs:4-44` has no `fLaC` magic branch. `CAPABILITIES.json` does not list `flac` under `containers` (lines 44-102) nor `vorbis_comment` under `namespaces` (lines 4-43). Parent issue #5 (kstore creator wedge) names FLAC as a P0 audio format for creators who deliver lossless masters; this sub-issue delivers phase 1: a native FLAC metadata-block parser (STREAMINFO / VORBIS_COMMENT / PICTURE), a reusable `xifty-meta-vorbis-comment` namespace crate (also consumed by OGG in Task 5), detection, CLI wiring, and capability advertisement. No FFI shape change.

## Approach
Mirror the container/metadata split established by MP3 (Task 3 sibling #53) and the other image formats:
- `xifty-container-flac` owns FLAC stream framing: the 4-byte `fLaC` magic, the chain of `METADATA_BLOCK_HEADER`s (last-flag + type + 24-bit big-endian length), and exposes typed slices for STREAMINFO (bounded audio parameters: sample_rate/channels/bit_depth/total_samples → duration), VORBIS_COMMENT (payload bytes forwarded to the metadata crate), and PICTURE (minimal decoded header: MIME type + width + height + depth, per FLAC PICTURE block spec which mirrors ID3v2 APIC layout).
- `xifty-meta-vorbis-comment` owns Vorbis-comment interpretation only: parse `vendor_length | vendor_string | user_comment_list_length | (length | "KEY=VALUE")*` framing (little-endian lengths per Vorbis-comment spec) into `MetadataEntry` values under a new `vorbis_comment` namespace, mapping well-known keys (`TITLE`, `ARTIST`, `ALBUM`, `DATE`, `GENRE`, `TRACKNUMBER`, `ALBUMARTIST`, `COMPOSER`, `COMMENT`) while preserving the raw `KEY=value` pair for unrecognized keys. This crate is reusable — the Task 5 OGG plan will call `decode_payload` from the OGG Vorbis/Opus/FLAC-in-OGG metadata packet.
- Surface FLAC-derived audio properties (`AudioSampleRate`, `AudioChannels`, `AudioBitDepth`, `DurationSeconds`) and PICTURE attributes (`PictureMimeType`, `PictureWidth`, `PictureHeight`) via a helper analogous to the MP3 `mp3_scalar_entry` at `crates/xifty-cli/src/lib.rs:954-970`. Because `crates/xifty-policy/src/lib.rs:218-238` normalizes `audio.channels`/`audio.sample_rate`/`audio.bit_depth` by tag name, the new FLAC entries feed the existing normalization map without policy changes.
- Detector reference: `crates/xifty-detect/src/lib.rs:22-28` (ID3/MPEG branch) is the pattern to mirror — `fLaC` is a 4-byte leading magic. CLI wiring reference: `Format::Mp3` arms at `crates/xifty-cli/src/lib.rs:71-74` (probe) and `crates/xifty-cli/src/lib.rs:467-472` + `mp3_entries` at lines 972-1020 (extract). Capability reference: `mp3` container block in `CAPABILITIES.json:97-101` and `id3v2` namespace at lines 35-42.

## New files
- `crates/xifty-container-flac/Cargo.toml` — mirror `crates/xifty-container-png/Cargo.toml`; depend on `xifty-core` and `xifty-source` only.
- `crates/xifty-container-flac/src/lib.rs` — `parse(&SourceBytes) -> Result<FlacContainer, XiftyError>` and `parse_bytes(&[u8]) -> Result<FlacContainer, XiftyError>`. `FlacContainer` carries `nodes: Vec<ContainerNode>`, `issues: Vec<Issue>`, a decoded `StreamInfo { sample_rate_hz: u32, channels: u8, bits_per_sample: u8, total_samples: u64, duration_seconds: Option<f64> }`, `vorbis_comment_payload: Option<(u64 /*offset_start*/, u64 /*offset_end*/, &[u8])>` exposed via an accessor, and `pictures: Vec<FlacPicture>` (`mime_type: String`, `width: u32`, `height: u32`, `color_depth: u32`, `offset_start`, `offset_end`). Walk the block chain: after 4-byte `fLaC` magic read 4-byte header (last-flag bit | 7-bit type | 24-bit BE length), emit one `ContainerNode` per block (labels `streaminfo`, `padding`, `application`, `seektable`, `vorbis_comment`, `cuesheet`, `picture`, `reserved`), and stop after the last-metadata-block flag. Emit `flac_malformed_block` `Issue` on truncation. Unit tests mirror `crates/xifty-container-png/src/lib.rs` synthetic-fixture style.
- `crates/xifty-meta-vorbis-comment/Cargo.toml` — mirror `crates/xifty-meta-iptc/Cargo.toml`; depend on `xifty-core` only.
- `crates/xifty-meta-vorbis-comment/src/lib.rs` — `pub struct VorbisCommentPayload<'a> { bytes: &'a [u8], container: &'a str, offset_start: u64, offset_end: u64 }` and `pub fn decode_payload(payload: VorbisCommentPayload<'_>) -> Vec<MetadataEntry>`. Parse Vorbis-comment framing (all lengths little-endian u32), split each comment on the first `=`, UTF-8-validate both sides, emit entries under `namespace: "vorbis_comment"` with `tag_id` = upper-cased key and `tag_name` matching one of the recognized list above (falling back to the raw key for unrecognized names). Surface the `vendor_string` as a `Vendor` entry. Emit nothing and swallow malformed tail bytes (caller is responsible for the outer `Issue`). Unit tests cover: minimal TITLE/ARTIST/ALBUM round-trip, unknown key passthrough, multi-value (repeated) keys, empty comment list, malformed length tail.
- `fixtures/minimal/happy.flac` — generated by `tools/generate_fixtures.py`: `fLaC` magic + STREAMINFO (44100 Hz, 2 channels, 16 bps, e.g. 44100 total samples → 1.0 s) + VORBIS_COMMENT (TITLE/ARTIST/ALBUM) + PICTURE (type 3 front cover, MIME `image/png`, a 1x1 PNG payload) + 1-byte frame sync. Real FLAC audio frames are not required — the metadata block chain is independent of the frame payload.

3. Scaffold `crates/xifty-container-flac` (Cargo.toml + src/lib.rs) with `parse_bytes` walking the block chain, STREAMINFO decoder (20-byte fixed layout: 16-bit min-blocksize, 16-bit max-blocksize, 24-bit min-framesize, 24-bit max-framesize, 20 bits sample-rate, 3 bits channels-1, 5 bits bits-per-sample-1, 36 bits total samples, 128-bit MD5), VORBIS_COMMENT slice exposure, and PICTURE decoder (32-bit type, 32-bit MIME length + MIME, 32-bit description length + description, 32-bit width, 32-bit height, 32-bit color depth, 32-bit colors-used, 32-bit data length + data) — `cargo test -p xifty-container-flac` passes for a synthetic three-block stream.

## Risks
- **STREAMINFO bit-packing off-by-one.** The 20/3/5/36-bit packing is spec-exact; a wrong shift silently yields implausible sample rates (e.g. 0 or > 655350). Guard with an `Issue::Warning` when the decoded sample-rate is 0 and cover in unit tests with the canonical 44.1 kHz stereo/16-bit pattern.
- **Vorbis-comment endianness.** All lengths are little-endian, which differs from FLAC block lengths (big-endian 24-bit). The combined endianness split inside a single file is the most likely source of bugs; make the split explicit in the code and assert in tests.
- **PICTURE payload size.** PICTURE blocks can be large (embedded JPEGs of several hundred KiB). The parser should record `offset_start`/`offset_end` without copying the image bytes; decoded attributes (MIME/width/height/depth) come from the fixed header only.
- **Capability generator.** If `tools/generate_capabilities.py` is authoritative, verify it produces the new `flac` container and `vorbis_comment` namespace entries without reordering existing keys; if it is not authoritative, hand-edit `CAPABILITIES.json` and note that in the PR body.
- **Namespace name collision.** The `"flac"` namespace (used for STREAMINFO / PICTURE scalar entries) is distinct from `"vorbis_comment"` (Vorbis tags). Both are emitted by this task; policy (`crates/xifty-policy/src/lib.rs:218-238`) matches by `tag_name` not namespace, so no policy changes are required, but reviewer should confirm that no cross-namespace conflict fires on the synthetic fixture.
- **Insta snapshot drift.** The reviewer must use `cargo insta review` to accept the new snapshots deliberately (per SRS §4 drift policy).
- **OGG forward-compat.** The Vorbis-comment crate will be re-consumed from Task 5 (OGG). Keep `VorbisCommentPayload.container` generic (a `&str`) so the OGG caller can pass `"ogg"` without crate-level coupling; add a doc-comment noting the planned reuse.
