# Plan #29: PNG and WebP IPTC namespace extraction

## Problem
`CAPABILITIES.json` marks `png.iptc` and `webp.iptc` as `not_yet_supported`, yet the IPTC IIM/Photoshop decoder (`xifty_meta_iptc::decode_payload`) is already wired for JPEG (`crates/xifty-cli/src/lib.rs:141-158`) and TIFF (`:234-253`). The missing piece is container-layer payload detection in `xifty-container-png` and `xifty-container-riff` and the corresponding CLI extract arms for `Format::Png` (`:256-321`) and `Format::Webp` (`:322-378`), plus fixtures and insta snapshots.

## Approach
Mirror the existing XMP/ICC routing in both containers. For PNG, expose an `iptc_payloads()` iterator over `iTXt`/`zTXt`/`tEXt` chunks whose keyword is `Raw profile type iptc` (the ImageMagick/ExifTool convention: hex-encoded IIM inside a text chunk) and, as a secondary route, any raw `iTXt`/`zTXt` with keyword `IPTC-NAA`. Add a small `decode_png_iptc_payload` helper in the CLI (alongside `decode_png_iccp_payload` at `crates/xifty-cli/src/lib.rs:451-462`) that strips keyword/compression framing and hex-decodes when `Raw profile type iptc` is detected, yielding raw IIM bytes for `decode_iptc_payload`. For WebP, IPTC is written by Adobe/ImageMagick as a `RIFF` chunk id `"IPTC"` (4-byte id) — add an `iptc_payloads()` iterator mirroring `icc_payloads()` (`crates/xifty-container-riff/src/lib.rs:34-38`). Route both through `xifty_meta_iptc::decode_payload` with provenance paths `png_iptc` / `webp_iptc`. Extend `tools/generate_fixtures.py` to produce `iptc.png` (zTXt with `Raw profile type iptc` keyword) and `iptc.webp` (RIFF `IPTC` chunk) using the existing `build_iptc_iim()` payload. Promote both container capabilities to `bounded` by running `python3 tools/generate_capabilities.py --write` (or hand-promote, since the generator preserves `bounded`).

## Risks
- PNG IPTC embedding is not formally standardized; two competing conventions exist: ImageMagick's `Raw profile type iptc` (hex-encoded in `zTXt`/`iTXt`/`tEXt`) and Adobe's `IPTC-NAA` binary keyword. Exiftool writes the ImageMagick form. Supporting both keeps us honest against real-world files; the plan covers both.
- WebP IPTC chunk id: the RIFF convention used by ExifTool/ImageMagick is a 4-byte `IPTC` chunk (no trailing space). If a fixture in the wild uses a different id, extraction will silently miss it — acceptable for the `bounded` promotion but worth calling out.
- The `zTXt` path requires zlib inflate; reuse the `ZlibDecoder` pattern from `decode_png_iccp_payload` (`:458`). `iTXt` with compression flag set is rare; handle it defensively or emit `png_iptc_payload_invalid`.
- `generate_capabilities.py --write` promotes `not_yet_supported` to `supported`, not `bounded`. To match the acceptance criterion (`bounded` or better), hand-edit or rely on the generator's "Preserve 'bounded' and existing 'supported' as-is" behavior — set `bounded` first, then run `--check`.
- Snapshot review must be manual per guardrails ("never blanket-accept").
